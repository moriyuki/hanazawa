・システムテスト仕様書の作成
・お客さんは何が欲しいかわかっていない。これを明確にする手助けをするための文書
・「◯◯する」「◯◯できる」という書き方より、「◯◯はしない」という書き方が大事。
・目的の共有を事前に行う
・コンセプトを決める。
・最初は「夢のアプリ」を書く。
・目的から外れた要件、機能を除外する（「◯◯はしない」と記載）
・小さいゴールのための機能仕様書(1)〜(5)、(2)〜(5)は流動的

テーマ：
- Windows Universal Application
REST API を使ってみる
仕様書作成、テスト仕様書作成

目的：
自分の仕事を簡単に漏れなく管理したい。そんなツールが欲しい。
Windows端末はあるものとする。
既存のツールは設定箇所が多いので入力が大変でやだ。

要求：
- シンプルな操作性
- オフラインでも使える
- WindowsPhone対応
- 無料
- WindowsStoreからダウンロードできる
- 画面は、一覧と詳細。
- チケット追加すると詳細は適当な値で更新される
添付ファイルに画像を投稿できる
関係するチケットを視覚的に表現される
チケットは完了したらしばらく完了状態で表示し、適当に非表示にされる
日時設定は基本空。したければ細かく設定できる
メールを出すとチケット化したい。
※要求分析で、企画する場合はペルソナを作成することがある。ペルソナは、年齢性別、趣味趣向を細かく勝手に決めるもの。

要求定義：
シンプルな操作性(１行記録(チケットタイトルのみ)と完了記録のみにする。オプションで期限、詳細を記録できる)
操作で自動的に補間、入力は１行、チケットの内容がわかるように書いてもらう
例：「◯◯さんに▲▲のメールを送る」などは記入してもらう
複数のプロジェクトから、選択できるような仕組みを設ける。InBoxをデフォルトにし、必要に応じてプロジェクトを変更できる。
必要に応じて、期限を設定できる。
一覧で自分のチケットを確認できる。一覧で表示するチケットの数は画面サイズに依存する。一覧は古いものから順に表示する。一覧に表示するのは、自分のチケットのみ。
ショートカットでチケットを追加できる。
チェックボックスをチケットに付けて、１クリックで完了。
完了したチケットは完了状態がわかるように背景をグレーにして２４時間表示。
２４時間以内に完了の取り消しがなければ非表示。
非表示になったチケットは表示させない。
オフラインでも使える
ローカルにDBを持つ。オンライン時に裏で同期する。データの同期はユーザ操作とは非同期にする。同期処理にエラーが発生した場合はサーバのデータを正とし、ローカルDBを更新する。
初回起動時がオフラインの場合は警告画面を出す。まっさらな状態で表示。
コンフリクト(ローカルとサーバのデータ不一致)が発生した場合は、後勝ちにする
WindowsPhone対応
Universal　アプリで開発
無料アプリであること
広告表示について了解してもらう
WindowsStoreからアプリ利用可能
要調査
画面は一覧と詳細
